UofT

My last uoft lectures were almost exactly 3 years ago, so I thought it’d be really fun to look through my uoft monorepo. It’s just a single repository of all my assignments and labs for my courses at uoft.

It started on October 1, 2015, which just about lines up with the first day of the first assignment of the first programming class. I don’t remember why I started it, but I’m still profiting off of it 7 years later, so I should probably do this more often when starting new things.

Learnings and stuff

  1. About 2 years into it, I started prefixing commit messages with both the course and the assignment when applicable, for example:

    $ git show 337cec38bcccbb210138b91de5aa65a58caae80f
commit 337cec38bcccbb210138b91de5aa65a58caae80f
Author: Kashav Madan <kshvmdn@gmail.com>
Date:   Sun Oct 22 11:39:55 2017 -0400

    csc324: a1: add starter code

    I think I noticed some large open source monorepos used that pattern and I liked it.

  2. Per-directory gitignores only apply to the interstitial directory tree, which I thought was cool. This was particularly useful because all courses were also version controlled on separate school-hosted git or svn servers, so I didn’t have to duplicate gitignores or worry about pushing code to the wrong server.

  3. Commit early, commit often.

    On Jan 29 2019 at 1:19 pm, I made this commit for csc367 to add the starter code for an assignment and some labs:

    $ git show 40499f752b4995b69ef2b50b8cf44953a1e04840
commit 40499f752b4995b69ef2b50b8cf44953a1e04840
Author: Kashav Madan <kshvmdn@gmail.com>
Date:   Tue Jan 29 13:19:21 2019 -0500

    csc367: lab01,lab02,a1: add starter code and lab implementations

    On Jan 29 2019 at 1:44 pm, 25 minutes later, I made this commit to remove that course entirely (because I dropped it):

    $ git show cb8c3e0781c65b0d69b6c868f1eb3414c52b9a8c
commit cb8c3e0781c65b0d69b6c868f1eb3414c52b9a8c
Author: Kashav Madan <kshvmdn@gmail.com>
Date:   Tue Jan 29 13:44:52 2019 -0500

    csc367: remove course

    So I’m glad that history is still around.

  4. Do code reviews for yourself! I’m sure most people implicitly do this already, or learn to do it by the time they’ve been in industry for a few years, but I’ve always found a ton of value in pushing code early (after “reviewing” and “testing” it locally) and then looking through it later under a different light.

    My brain keeps documents and code paged into memory, which means I’m always subconsciously reviewing a thing even after I’ve left it alone, so OctoDroid was really helpful when I realized a bug or a nit or a better way to word things and didn’t have a computer nearby.
